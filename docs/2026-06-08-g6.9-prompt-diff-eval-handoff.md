## Why this exists

After G5/G6.1b, the eval suite measures the two things Em does *defensively*: her **verdicts** (precision 0.97 / recall 1.00 / false_pass 0.00) and her **citations** (cites-correct 0.97). It measures **nothing** about her *constructive* output — the `proposed_patches` she stages into `manifest.lock.yaml`. Per the Critic Stack contract, T2 "**proposes prompt diffs, not pass/fail**." "Flags correctly" and "proposes a fix that helps" can diverge arbitrarily: Em can catch every defect and propose diffs that are useless, harmful, or ungrounded. G6.9 builds the second axis of the ruler.

The hard dependency — citation grounding at cites-correct 0.03 — is **CLEARED** (0.97, ratified + merged #29, criteria-text ON in production). Diff *grounding* is now scoreable against signal, not noise.

## Gate 0 — the data-path wiring (prerequisite, $0, do this FIRST)

**The diffs are not captured by the eval harness today.** Verified 2026-06-08:
- `AgentResult.proposed_patches: list[Patch]` IS populated by Em, and `Patch` carries `target / path / operation / value / rationale / proposed_by / cites_criteria` (`pipeline/agents/__init__.py`).
- The **production** DAG path captures them: `patch_stager.py` → `runs/{run_id}/manifest.lock.yaml`; `python -m pipeline.cli patches list` reads them.
- The **eval** path **discards** them: `CaseScore` (`evals/vision_critic/scoring.py:93`) persists `expected_verdict / predicted_verdict / expected_cites / actual_cites / confidence / wall_s / reasoning` — **no `proposed_patches` field**. `score.py` never threads `result.proposed_patches` into the CaseScore.

**Wiring task (deterministic, no model spend):**
1. Add `proposed_patches: list[Patch]` (and the derived per-diff `cites_criteria`) to `CaseScore`.
2. Thread `result.proposed_patches` through the per-case scoring path in `score.py` (the place that builds CaseScore from the AgentResult).
3. Give the `conftest.py` / `score.py --stub` defect-case stub a **non-empty** `proposed_patches` example so CI exercises the capture (today both stubs return `[]`).
4. Surface captured diffs in the trace's per-case detail block (extend the existing G6.2 cite+reasoning table or add a sibling).

**Verify end-to-end before declaring Gate 0 done** — not just that the field exists:
- Does the eval runner pass `result.proposed_patches` to the CaseScore builder (or is it dropped at the call site)?
- Does a real (non-stub) defect-case run actually populate diffs? Run one case live, confirm the trace shows a captured patch.
- Does the capture survive the N=5 majority collapse in `consensus()` (pick patches from the run whose verdict won, like cites already do)?

## Gate 1 — golden-diff corpus authoring (Sean, critical path)

Each defect case in `evals/vision_critic/cases.yaml` gains a `golden_diff` field: the reference prompt-clause Sean would actually add to fix that defect (e.g. palette defect → the exact corrective clause). **~30 cases — Sean's authoring, the only thing on the critical path.** This blocks Gate 2 scoring. Author across all six single-axis classes so the per-class segmentation is populated.

Discipline carried from the corpus build: **ships-red is absolute** — a golden diff describes the *fix*, never relaxes the defect label. Golden diffs are reference fixes, not answer keys Em is tuned toward.

## Gate 2 — Design 2 scorer (golden-diff agreement, the in-suite gate)

- Per defect case: judge-scored **semantic match** between Em's `proposed_patches[].value` and the case `golden_diff`. Binary per case.
- **Cite scoring (locked #4): precision AND recall on the diff's `cites_criteria`** — credit the diff for citing the right rule, penalize citing irrelevant ones. Distinct from the recall-only verdict scorer; build it as a separate scoring axis, don't reuse the verdict cite-matcher wholesale.
- **Judge calibration is mandatory** per the eval handbook ([`docs/research/2026-05-31-ai-evals-best-practices-and-anima-eval-strategy.md`](research/2026-05-31-ai-evals-best-practices-and-anima-eval-strategy.md)): calibrate the agreement judge against a Sean-labeled sample before trusting it; record the judge-bias ledger.
- Known limitation to document, not solve: golden agreement penalizes valid *alternative* fixes (a diff can disagree with the golden one and still work). Design 1 is what catches that — that's why it's layered, not standalone.
- Runs in-suite, uncosted at eval time after authoring. Segmented per-class.

## Gate 3 — Design 1 periodic empirical loop (the fix-rate anchor)

The empirical anchor, the closest usefulness has to ground truth (aligns with Engine Truth). **Periodic + sampled, not per-run (locked #6); gated behind §0 (Gate 0/§0).**
- Per sampled defect case: capture Em's diff → apply it → re-generate the frame (content-hashed node) → re-critique with Em (or the deterministic T1 gate where the defect class is deterministic). Binary: **defect cleared AND identity/style held**.
- Confounded by generation stochasticity → **N re-rolls per diff, report clear-rate with a band**, exactly like the N=5 verdict band. A diff that "works" once may have worked by accident.
- Dated runs like the verdict baselines (`evals/vision_critic/traces/`). Headline = fix-rate.
- **Unit cost ≈ 1 NB2 generation + 1 Em call per case per re-roll.** Bounded `GEMINI_API_KEY` from `.env`; size the sample + N to the budget and record the estimate in the field report before the run.

## Gate 4 — Design 3 triage rubric (judge, never headline)

LLM judge on a binary rubric per diff: targets the cited defect / actionable as a prompt edit / doesn't re-describe the character (Seedance rule) / doesn't violate the style-neutrality doctrine ([`docs/prompt-style-neutrality-doctrine.md`](prompt-style-neutrality-doctrine.md)). Calibrate against a Sean-labeled sample first. **Triage signal only — judge-on-judge is the weakest evidence class; never the headline number.** Cheap, runs on existing traces; worth having early as a dry read on the already-persisted reasoning while Gate 1 authoring is in flight.
