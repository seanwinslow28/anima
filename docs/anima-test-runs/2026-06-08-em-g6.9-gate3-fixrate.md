## 1. The Gate-2 v1 crash (and why the runbook over-claimed)

The first launch (`both+null 12×3`) **crashed on case #0** (`anatomy-ad1-six-fingers`, identity_critical, sampled first) after ~909s ≈ exactly the 900s `_PER_CASE_TIMEOUT_S`. The `subprocess.TimeoutExpired` raised by `_run_case_subprocess` was **uncaught** — it propagated out of the orchestrator loop and aborted the whole run, discarding every in-memory result. **0/12 cases completed; the JSON was empty.**

The runbook §0 claimed the live path "self-isolates — a worker that dies before emitting its sentinel is an honest errored gap (`RuntimeError`), not a silent zero." Verified against the tree, that was **false**: Stage 1 (#37) isolated the *worker* from the exit-144 teardown race, but the **orchestrator loop (patch_efficacy.py:617-619) had no try/except** — both the `TimeoutExpired` and the `RuntimeError` paths aborted the run. (Standing doctrine — verify the tree, never trust a label, including the runbook — caught it.)

Sunk spend: a partial single case + Gate 1 ($0.05) — well under ~$1 (exact call counts lost with the worker's captured output on the timeout). State after crash: orphan sweep clean, baseline guard byte-identical, no tracked changes.

## 2. The containment fix (TDD)

`evals/vision_critic/patch_efficacy.py`: extracted the live per-case loop into **`_run_live_cases`** — a per-case `try/except` that records a `TimeoutExpired` or any worker `Exception` as an **honest errored gap `(name, reason)`** and **continues**, mirroring score.py's containment (score.py:660-672). Output JSON gained an `errored: [...]` list; a stderr NOTE names gapped cases. The stale `COSTED-RUN HARDENING NOTE` docstring (predating #37) was corrected to describe the now-wired isolation.
