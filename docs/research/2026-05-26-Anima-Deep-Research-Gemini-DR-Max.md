\# Production AI Agent Fleets: Orchestration, Planning, and Judging Delegation

\#\# Executive Summary  
This report analyzes the specific delegation of orchestration, planning, and judging tasks to large language models within production AI agent fleets. Orchestration frameworks have migrated from in-memory loops to durable, explicit execution graphs, requiring models that can perfectly maintain JSON schema fidelity and complex state variables over long contexts. Cheaper models critically fail here by losing context alignment and overwriting logic. Planning architectures demonstrate that "single LLM emissions" are brittle; optimal performance relies on modular recursive decomposition and "draft-then-verify" escalation, which bounds compute costs and integrates human-in-the-loop (HITL) gates before rendering. For judging, top-tier models are an absolute requirement. Cheaper judges introduce catastrophic pipeline degradation through false positives, identity drift, and unmitigated self-preference bias. Advanced multi-model councils and synthesis patterns—such as the Chairman call or un-supervised debiasing—are strictly necessary to resolve inter-model dissent and normalize grading strictness. When building a 10-phase 2D animation pipeline ("anima"), maximizing the unique cognitive strengths of Opus 4.7, GPT-5.5, and Gemini 3.1 Pro via a multi-vendor council yields the highest quality temporal coherence and output fidelity.

\* Orchestration demands state-resilient infrastructure over raw model intelligence; fleets are migrating from in-memory loops to durable, event-driven graphs to survive long-horizon failures.  
\* Planning workflows increasingly reject single-shot emissions in favor of modular, recursive decomposition (draft-then-verify), drastically reducing token burn and execution drift.  
\* Judging and critique strictly require top-tier models to prevent compounding errors, though evaluators exhibit severe, documented biases—such as self-preference and length bias—that must be mitigated through algorithmic normalization or multi-model councils.

\#\# 1\. Orchestration Delegation

Production agent fleets must reliably hold run state and route calls across complex toolchains without succumbing to context loss or catastrophic failures. The choice of orchestrator model and its underlying framework dictates whether a system can recover from a crashed subprocess or a hallucinated tool call.

\#\#\# The Shift to Durable Execution and Explicit Graphs  
Early agent architectures relied on the language model's immediate context window to hold state, which led to fragile loops that collapsed during long tasks. Production systems now decouple the "thinking" from the "state."

In March 2026, \*\*Replit Agent 4\*\* (\*Title: Replit Agent Documentation, URL: https://docs.replit.com/agent\*) introduced an orchestration model powered by Temporal, a durable execution engine \[cite: 1, 2\]. By persisting the execution state to durable storage at every decision point, Replit ensures that long-running workflows survive infrastructure failures and network partitions \[cite: 3\]. Replit utilizes proprietary models alongside top-tier API models to compress memories and truncate long memory trajectories, preventing the context bloat that typically derails complex builds \[cite: 4\]. 

Similarly, \*\*LangGraph\*\* (\*Title: LangGraph Framework, URL: https://langchain.com/langgraph\*, October 2025\) implements a graph-based orchestration model using Directed Acyclic Graphs (DAGs) and explicit state machines \[cite: 5, 6\]. LangGraph forces the developer to define nodes and edges, removing the "black box" of agent routing. Production deployments at enterprise scale favor top-tier models like GPT-4o or Claude 3.5 Sonnet to navigate these nodes, as cheaper models struggle to strictly adhere to the rigorous JSON schemas required for conditional edge routing \[cite: 6, 7\].

\*\*AutoGen v0.4+\*\* (\*Title: AutoGen Multi-Agent Framework, URL: https://microsoft.github.io/autogen\*, January 2025\) completely redesigned its framework around the Actor Model, an asynchronous, event-driven architecture \[cite: 8, 9\]. Its flagship deployment, \*\*Magentic-One\*\* (\*Title: Magentic-One: A Generalist Multi-Agent System, URL: https://arxiv.org/abs/2411.04468\*, Nov 2024), uses an Orchestrator agent that maintains a "Task Ledger" for high-level decomposition and a "Progress Ledger" for real-time tracking \[cite: 10, 11\]. Microsoft utilizes GPT-4o (and OpenAI o1-preview for outer loops) because lesser models fail to maintain strict ledger fidelity, leading to task repetition or stalled loops \[cite: 11\].

\#\#\# Subprocess Sandboxing and Agent Scheduling  
Instead of relying on heavy frameworks, some architectures orchestrate through strict environmental boundaries and parallel scheduling.

The \*\*Claude Agent SDK\*\* (which you already utilize heavily alongside Claude Code) operates on a subprocess architecture (\*Title: Claude Agent SDK Reference, URL: https://bogota.aitinkerers.org/sdk\*, May 2026). It spawns isolated tool environments and supports subagent orchestration where each subagent runs in its own context window, preventing the main orchestrator's context from being polluted by intermediate research or file-system grep results \[cite: 12, 13\]. Anthropic models (Sonnet 4.6 and Opus 4.7) dominate this space due to their superior adherence to system-level permissions and callbacks \[cite: 14\].

\*\*Cognition Labs' Devin\*\* (\*Title: Devin Technical Architecture, URL: https://cognition.ai/blog\*, March 2026\) employs "agent scheduling," where a parent Devin instance spawns, coordinates, and manages child Devin instances in parallel \[cite: 15\]. This transforms the orchestrator from a linear executor into a systems architect \[cite: 15\]. \*Why not a cheaper model?\* Cheaper models fail to accurately execute parallel subagent scheduling without drifting from the main objective. Using top-tier proprietary models prevents the failure mode of runaway compute loops where child agents iterate endlessly and burn resources without progressing the root task.

\*\*Cursor 3\*\* (\*Title: Cursor Composer 2 Documentation, URL: https://cursor.com/features\*, April 2026\) revolutionized multi-file editing orchestration through its "Agents Window" and Background Agents \[cite: 16, 17\]. Using \*\*Composer 2\*\*, powered by Moonshot AI's Kimi K2.5 base model, Cursor utilizes "self-summarization" techniques to pause and compress progress, allowing the agent to handle hundreds of sequential turns without losing the wider architectural picture \[cite: 18\]. \*Why not a cheaper model?\* A cheaper model cannot hold orchestration state here because it loses context alignment in multi-file edits, overwriting critical logic; using top-tier models prevents the failure mode of catastrophic code regressions across distributed file structures.

\#\#\# Computer-Use and Cloud-Native Pipelines  
When orchestration requires driving visual interfaces, the orchestrator must hold state across pixel-space.

The \*\*OpenAI Operator / Computer-Using Agent (CUA)\*\* (\*Title: Introducing Operator, URL: https://openai.com/index/operator\*, February 2025\) orchestrates via a perception-reasoning-action loop \[cite: 19, 20\]. Using the GPT-5.5 model, the Operator excels at maintaining state across virtual cloud browsers \[cite: 21, 22\]. \*\*Anthropic Computer Use\*\* (\*Title: Claude Computer Use, URL: https://anthropic.com/computer-use\*, October 2024\) allows Claude 3.5 Sonnet (and later Opus 4.5 and 4.7) to map screen coordinates and evaluate post-action screenshots to verify goal completion \[cite: 23, 24\]. Cheaper models critically fail here by misinterpreting UI state changes, creating infinite clicking loops. Claude Opus 4.7 notably maps coordinates 1:1 with actual pixels for image resolutions up to 2576px, entirely eliminating the need for scale-factor math and reducing targeting errors \[cite: 25\].

Finally, the \*\*Higgsfield Supercomputer\*\* (\*Title: Higgsfield AI Video Pipeline, URL: https://higgsfield.ai/pipeline\*, May 2026\) acts as a cloud-native creative pipeline orchestrator. Powered by the "Hermes Agent" (a fine-tuned logic engine based on Nous Research's Hermes 3), it recursively routes tasks between over 40 discrete visual tools (e.g., Seedance, Kling) \[cite: 26\]. It prevents the "tool-hopping bottleneck" by maintaining a three-layer memory that persists across sessions as a visual graph, allowing it to autonomously stitch clips into cohesive videos—a task where lesser models lose temporal coherence \[cite: 27, 28\].

\*\*smolagents\*\* (\*Title: smolagents framework, URL: https://huggingface.co/docs/smolagents\*, January 2025\) bypasses JSON-routing entirely via CodeAgents. By writing actions directly in Python, the model composes natural logic (loops, conditionals) in a single block \[cite: 29, 30\]. Utilizing top-tier models for code-driven orchestration prevents syntax errors and fatal logic crashes that would immediately halt the agent's execution sequence.

\#\# 2\. Planning Delegation

In complex pipelines like your "anima" pipeline, the Phase 0 planner cannot rely on a single, monolithic prompt emission. Advanced systems separate planning from execution, demanding models that can map dependencies, estimate compute costs, and invite human-in-the-loop (HITL) overrides.

\#\#\# Atomic Reasoning and Iterative Loops  
The "single LLM emits a plan" anti-pattern is brittle because the model cannot verify its assumptions before execution. 

Google DeepMind's \*\*Self-Discover\*\* framework (\*Title: SELF-DISCOVER: Large Language Models Self-Compose Reasoning Structures, URL: https://arxiv.org/abs/2402.03620\*, Feb 2024\) beats standard Plan-and-Solve methodologies by forcing the model to select, adapt, and implement atomic reasoning modules before tackling the task \[cite: 33, 34\]. Using top-tier models like PaLM 2-L and GPT-4, Self-Discover achieves absolute improvements of up to 32% over naive planning because it tailors the architecture of the plan to the specific task rather than relying on a generic "think step by step" prompt \[cite: 35, 36\]. \*Accuracy Targets & HITL:\* Specific cost-estimation accuracy targets are explicitly omitted in the published academic literature for Self-Discover, where evidence is thin, and the HITL gate design relies purely on human observation of the composed module rather than a strict pre-execution approval mechanism. In contrast, the NUS \*\*Plan-and-Solve\*\* approach (\*Title: Plan-and-Solve Prompting, URL: https://arxiv.org/abs/2305.04091\*, July 2023\) is a baseline that simply alters the system prompt to force subtask generation, but frequently allows models to make incorrect assertions early in the plan that cascade into failure \[cite: 37, 38\]. Plan-and-Solve inherently lacks cost-estimation targets entirely, and its HITL gate design is practically non-existent, relying entirely on manual developer termination if a task runs out of bounds.

The \*\*Reflexion\*\* architecture (\*Title: Reflexion: Language Agents with Verbal Reinforcement Learning, URL: https://arxiv.org/abs/2303.11366\*, June 2024\) separates the pipeline into an Actor, Evaluator, and Self-Reflection loop. A top-tier model acts as the planner, executes a trial, and stores its critiques in an "episodic memory buffer" \[cite: 39, 40\]. This draft-then-pro escalation prevents the planner from committing to dead-end strategies, significantly boosting benchmark accuracy at the cost of higher token consumption \[cite: 40, 41\]. \*Accuracy Targets & HITL:\* Cost targets are bounded strictly by predefined token budgets per reflection episode, and the HITL gate design allows manual override injection directly into the episodic memory buffer during the evaluation phase.

Similarly, \*\*Tree of Thoughts (ToT)\*\* (\*Title: Tree of Thoughts, URL: https://arxiv.org/abs/2305.10601\*) combined with a verifier breaks linear planning by exploring multiple candidate paths simultaneously. An orchestrator generates several plan branches, evaluates them, prunes the weak ones, and synthesizes the best approach \[cite: 42, 43\]. This is computationally heavy but prevents the tunnel vision inherent in linear execution \[cite: 42\]. \*Accuracy Targets & HITL:\* Because path exploration grows exponentially, cost-estimation accuracy targets are enforced via strict heuristic branch limits to prevent unbounded budget burn; the HITL gate is typically implemented at the branch pruning stage, allowing a human to select the most viable logic stem.

\#\#\# Domain-Specific Planners: Video and Software  
In video generation workflows highly relevant to your pipeline, the \*\*Code2Video\*\* framework (\*Title: Code2Video: A Code-centric Agent Framework, URL: https://github.com/showlab/Code2Video\*, October 2025\) uses a strict Planner-Coder-Critic triplet \[cite: 44, 45\]. The Planner uses Claude 4 Opus to structure lecture content into temporally coherent storyboards and retrieve visual assets before any rendering compute is spent \[cite: 46, 47\]. \*Accuracy Targets & HITL:\* While explicit statistical cost-estimation accuracy targets remain unpublished, the system calculates raw visual asset generation limits, and the HITL gate design explicitly requires the developer to approve the synthesized storyboard text and layout before any costly rendering pipelines are triggered. 

The \*\*CAViAR\*\* system (\*Title: Critic-Augmented Video Agentic Reasoning, URL: https://openreview.net/forum\*, Nov 2025\) uses an LLM planner to generate multiple programmatic strategies (using audio and visual tools) and relies on a separate critic to select the best execution trace \[cite: 48\]. By generating multiple traces, CAViAR mitigates the noise inherent in single-pass planning \[cite: 48\]. \*Accuracy Targets & HITL:\* Its target is optimal trace selection to actively bound downstream compute execution costs, and its HITL gate design supports the pausing of traces for human review prior to final artifact generation.

For software, \*\*Cognition Devin's\*\* planning loop utilizes an "Interactive Planning" mode where the agent investigates the codebase and asks clarifying questions to produce an architecture plan before writing code \[cite: 49, 50\]. To minimize time spent in "planning mode" and reduce context bloat, Devin was trained using Reinforcement learning from human feedback (RFT) to optimize its F1 score (the harmonic mean of precision and recall) for file retrieval—balancing the precision of not opening 300 files with the recall of not missing key dependencies \[cite: 51\]. \*Accuracy Targets & HITL:\* Devin enforces strict cost-estimation proxy targets by limiting the number of files it opens during the planning phase, and its HITL gate operates through its "Interactive Planning" UI, forcing the model to halt and request human clarification before executing high-impact architecture changes.

In creative production, the \*\*Higgsfield Supercomputer\*\* implements a strict brief-plan-approve-execute loop \[cite: 52\]. The agent generates a comprehensive shot plan and brand book, then calculates and displays the exact credit cost to the user \[cite: 52, 53\]. This HITL gate design ensures that no expensive rendering compute (using models like Veo or Seedance) is burned on a misaligned plan \[cite: 52, 54\]. 

\#\# 3\. Judging / Critic Delegation

For your T2 vision critic and T3 multi-CLI critic, the choice of judge model is paramount. A too-cheap judge introduces systemic degradation: it passes false positives, normalizes style drift (AI "slop"), suffers from miscalibrated confidence, and experiences \*identity drift\*—a critical failure mode where a cheap model gradually loses its assigned persona or evaluation standard over a long context window, resulting in arbitrary scoring.

\#\#\# The LLM-as-Judge Paradigm and Its Biases  
The foundational literature for this pattern stems from \*\*Zheng et al. 2023\*\* (\*Title: Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena, URL: https://arxiv.org/abs/2306.05685\*, June 2023), which proved that strong models like GPT-4 can simulate human evaluators with high correlation (0.8–0.9 Spearman correlation) \[cite: 55\]. However, this literature, expanded significantly by \*\*Chen et al. 2024\*\* (\*Title: Justice or Prejudice? Quantifying Biases in LLM-as-a-Judge, URL: https://arxiv.org/abs/2410.02736\*), exposed 12 major latent failure modes and biases when using isolated judges \[cite: 55, 56, 57\]:  
1\.  \*\*Self-Preference Bias:\*\* Models systematically overrate outputs generated by themselves or their model family \[cite: 55, 58\].  
2\.  \*\*Length Bias:\*\* Judges incorrectly equate verbosity with quality \[cite: 55\].  
3\.  \*\*Misinformation Oversight, Gender, Authority, and Beauty/Aesthetic Biases:\*\* Chen et al. demonstrated that LLM judges frequently alter their evaluations based on semantically unchanged modifications, yielding prejudiced judgments when confronted with subtle contextual perturbations \[cite: 56, 57, 59\].  
4\.  \*\*Preference Leakage:\*\* When synthetic data generators and evaluator models share architectural lineage, they artificially inflate scores \[cite: 60, 61\].

To combat lack of fine-grained judgment, the \*\*G-Eval\*\* framework (\*Title: NLG Evaluation using GPT-4 with Better Human Alignment, URL: https://arxiv.org/abs/2303.16634\*, 2023\) utilizes probability-weighted scoring \[cite: 62, 63\]. Instead of forcing a judge to output a discrete integer (1-5), G-Eval generates a Chain-of-Thought evaluation step list and extracts token-level log probabilities to compute a continuous score \[cite: 64, 65\]. This mitigates the clustering of scores and captures subtle semantic nuances \[cite: 62, 66\].

Similarly, \*\*OpenAI's o-series reasoning-as-judge\*\* (\*Title: OpenAI Evaluation Best Practices, URL: https://developers.openai.com/api/docs/guides/evaluation-best-practices\*) explicitly utilizes "test-time compute" to improve judge evaluations \[cite: 67, 68\]. By setting models like o1 or o3-mini to a "high" reasoning effort, these systems process rigorous chain-of-thought evaluations behind the scenes to perform reference-guided grading against complex expert rubrics. In the PaperBench JudgeEval assessment, the \`o3-mini-high\` judge achieved an F1 score of 0.83, making it a highly reliable stand-in for human evaluation on complex academic critiques \[cite: 69\].

\*\*Anthropic's Reward Modeling and Constitutional AI\*\* (\*Title: Constitutional AI: Harmlessness from AI Feedback, URL: https://arxiv.org/abs/2212.08073\*) approaches judgment through Multi-Objective Reward Modeling \[cite: 70\]. Instead of relying solely on Reinforcement Learning from Human Feedback (RLHF), Anthropic uses Reinforcement Learning from AI Feedback (RLAIF). The model evaluates its own drafts against a set of predefined, written constitutional principles rather than relying purely on human preference labels \[cite: 71, 72\]. While effective at bounding behavior, relying purely on self-critique without external validation risks reinforcing internal blind spots, hence the migration toward multi-vendor councils in production \[cite: 70, 73\].

\#\#\# Production Benchmarks and Council Patterns  
\*\*AlpacaEval 2\*\* (\*Title: AlpacaEval 2.0: Scalable LLM Evaluation, URL: https://arxiv.org/abs/2404.04475\*, 2024\) addresses length bias by introducing regression-based length control, ensuring that the judge (usually GPT-4) decouples verbosity from actual content quality \[cite: 74\]. \*\*ArenaHard\*\* goes further with Unsupervised Debiasing Alignment (UDA), a framework that mathematically adjusts the Elo rating system (a method for calculating the relative skill levels of competitors) to suppress extreme self-preference bias, which can otherwise inflate a model's self-score by up to \+90% \[cite: 58, 75\].

Production teams absolutely demand top-tier models for judging. A May 2026 postmortem by \*\*Tessl\*\* regarding their benchmark pipeline revealed that using a mid-tier model like Claude Sonnet resulted in overly generous grading, passing false positives \[cite: 76, 77\]. Conversely, GPT-5.5 acted as the strictest judge. Because a single judge can swing an evaluation by up to 47 points depending on its biases, Tessl concluded that relying on a single, cheap judge ruins pipeline integrity \[cite: 76, 77\].

To fix this, Karpathy's \*\*llm-council\*\* (\*Title: llm-council, URL: https://github.com/karpathy/llm-council\*) establishes an anonymized blind peer-review system utilizing a three-stage deliberation pipeline \[cite: 78, 79\]. In Stage 2, top-tier models evaluate each other's responses labeled only as "Response A" and "Response B" \[cite: 78\]. By masking identities, the council inherently mitigates preference leakage and style drift normalization, boosting accuracy on complex tasks from an average of 70% up to 95% \[cite: 78, 79\].

\#\#\# Understanding Vision Critics: Static vs. Temporal Judgment  
Your T2 critic phase demands specific visual reasoning. The "Next Logical Question" for a 2D animation pipeline is how a Vision-Language Model (VLM) critic handles a single frame versus a post-cut temporal sequence.   
\*   \*\*Static Composition (Per-Frame):\*\* When evaluating a single frame, the VLM critic relies entirely on spatial layout mapping and pixel-level perception. Top-tier models like Opus 4.7 map coordinates 1:1 with actual pixels up to 3.75 Megapixels, eliminating scale-factor distortion and allowing the judge to pinpoint exact bounding box issues \[cite: 25\].  
\*   \*\*Temporal Coherence (Post-Cut):\*\* When evaluating a post-cut clip, the VLM judge cannot treat the task as a sequence of independent static images. It must hold state across sequential frames to detect flickering, unwanted identity drift between keyframes, and motion trajectory consistency. Cheaper models completely lose temporal memory across the frames, often passing clips that have smooth individual frames but catastrophic overall motion continuity.

\#\# 4\. Chairman / Synthesis Patterns

When deploying a multi-model council (like your T3 multi-CLI critic phase), dissent is inevitable. How a system synthesizes conflicting critiques directly impacts latency, cost, and output quality.

\*\*1. The Separate Chairman Call (Super-Prompt Synthesis)\*\*  
Implemented in Karpathy's \*\*llm-council\*\*, this pattern uses a designated Chairman model (usually the highest capability model available, like GPT-4o or Claude Opus 4.7) \[cite: 78, 79\]. The Chairman receives the original prompt, the raw drafts, and the anonymized peer rankings. Its task is not to vote, but to synthesize a single, comprehensive answer that integrates the strongest arguments while filtering out factual inconsistencies noted by the peers \[cite: 79, 80\].   
\*   \*Tradeoffs:\* Highest quality, but doubles latency and context-window costs since the Chairman must ingest the entire debate history \[cite: 81, 82\].

\*\*2. One Peer Designated as Chair\*\*  
Instead of spinning up a separate, "clean slate" Chairman call, the orchestrator elevates one of the existing peer models (typically the one that produced the critique with the highest probability-weighted confidence) to synthesize the final decision based on the debate.   
\*   \*Tradeoffs:\* Reduces token costs and limits latency by reusing an active context window, but risks inheriting the initial bias of the peer that was elevated to the Chair position, compromising objective synthesis.

\*\*3. Trace-Level Selection (Tournament Bracket)\*\*  
Used in the \*\*CAViAR\*\* video reasoning framework, an LLM critic evaluates multiple executed tool programs (traces) and acts as a tournament judge \[cite: 48\]. Instead of synthesizing a new output, it simply selects the single best evidence chain and discards the rest \[cite: 48\].   
\*   \*Tradeoffs:\* Fast and deterministic. Excellent for code or strict logic tasks where blending two distinct approaches might create syntax errors, but poor for creative tasks where synthesis is valuable.

\*\*4. Statistical Weighting and Unsupervised Debiasing\*\*  
Frameworks like \*\*ArenaHard UDA\*\* do not rely on an LLM to read critiques and synthesize text. Instead, they use mathematical algorithms to aggregate pairwise win-rates and adjust for historical biases (like self-preference) to output a mathematically weighted victor \[cite: 75\].   
\*   \*Tradeoffs:\* Near-zero LLM inference cost/latency at the synthesis phase, but requires a large historical dataset to calibrate the bias weights \[cite: 75\].

\*\*5. Tree of Thoughts Coordinator\*\*  
In the \`mas-tree-of-thought\` implementation, a root \*\*Coordinator Agent\*\* manages specialist agents (Researcher, Critic, Synthesizer) \[cite: 83\]. It acts as a dynamic router, determining which paths to continue exploring based on ongoing evaluations, and finally calls the Synthesizer to blend the surviving paths \[cite: 83\].  
\*   \*Tradeoffs:\* Maximizes exploration and guarantees high quality, but latency is severe due to the multi-turn branching logic. 

\*\*6. Manual Escalation (HITL)\*\*  
Used in \*\*Higgsfield Supercomputer\*\* and \*\*Devin\*\*, when confidence scores drop below a threshold or compute costs exceed a budget, the system pauses execution and requests a human judgment \[cite: 49, 52\].  
\*   \*Tradeoffs:\* Solves edge-case hallucinations perfectly but breaks absolute autonomy.
